# 테스트 코드 작성 컨벤션

React Testing Library 기반 테스트(`*.test.tsx`) 작성 시 적용하는 규칙이다.

## BDD 스타일 구조

사용자 행동과 기대 결과를 중심으로 `describe` → `it` 구조를 구성한다.

### `describe` 사용 기준

`describe`는 테스트의 **맥락(context)을 구분**할 필요가 있을 때만 사용한다.

| 단계 | 목적 | 필수 여부 | 서술 방식 |
|------|------|----------|----------|
| 1단계 | 테스트 대상 (컴포넌트, 훅, 유틸) | 필수 | `describe('CartItem', ...)` |
| 2단계 | 독립적인 기능 그룹 | 선택 | `describe('수량 변경', ...)` |
| 3단계 | 조건 분기 | 선택 | `describe('수량이 최댓값일 때', ...)` |

**2단계를 사용하는 경우:**
- 테스트 대상 안에 독립적인 기능이 2개 이상일 때
- 각 그룹이 서로 다른 setup이나 컨텍스트를 필요로 할 때

**3단계를 사용하는 경우:**
- 동일 기능이 상태나 조건에 따라 다르게 동작할 때
- `~일 때`, `~인 경우`처럼 조건을 서술한다

**사용하지 않는 경우:**
- 테스트가 3개 이하이고 모두 동일한 컨텍스트 → 최상위 `describe` 하나로 충분
- `it` 이름만으로 맥락이 명확 → 중첩 불필요
- 공유 setup이나 조건 분기 없이 시각적 정리만을 위한 그룹화 → 불필요

### 예시

```tsx
// 1단계: 테스트 대상
describe('CartItem', () => {

  // 2단계: 기능 그룹
  describe('수량 변경', () => {
    it('증가 버튼을 클릭하면 수량이 1 증가한다', () => { ... });

    // 3단계: 조건 분기
    describe('수량이 최댓값(99)일 때', () => {
      it('증가 버튼이 비활성화된다', () => { ... });
    });

    describe('수량이 최솟값(1)일 때', () => {
      it('감소 버튼을 클릭하면 삭제 확인 다이얼로그가 표시된다', () => { ... });
    });
  });

  describe('상품 삭제', () => {
    it('삭제 버튼을 클릭하면 확인 다이얼로그가 표시된다', () => { ... });
    it('확인을 누르면 상품이 목록에서 제거된다', () => { ... });
  });
});
```

## `it` 작성 원칙

테스트 이름은 **유저 스토리**로 작성한다.
구체적인 구현(함수명, 메서드명, 내부 상태 변수명)을 테스트 이름에 포함하지 않는다.
구현 세부사항이 들어가면 내부 코드가 변경될 때마다 테스트 이름까지 함께 수정해야 한다.

```
// Bad — 구현에 결합된 테스트 이름
it('handleQuantityChange()가 호출되면 state가 업데이트된다')
it('setIsOpen(true)가 실행되면 모달이 열린다')
it('fetchCartItems가 성공하면 items 배열에 저장된다')

// Good — 유저 스토리 기반 테스트 이름
it('증가 버튼을 클릭하면 수량이 1 증가한다')
it('장바구니 아이콘을 클릭하면 장바구니 모달이 열린다')
it('페이지에 진입하면 장바구니 상품 목록이 표시된다')
```

- **사용자 행동 → 기대 결과** 형태로 작성한다.
- 테스트 이름만 읽어도 기능 명세서처럼 이해할 수 있어야 한다.
- `describe` 이름과 `it` 이름을 이어 읽었을 때 하나의 문장이 되도록 작성한다.

## 테스트 시나리오 책임 분리

각 테스트 파일은 **하나의 관심사**만 검증한다. 동일한 기능이라도 검증 대상에 따라 테스트 파일과 방식을 분리한다.

### 컴포넌트 렌더링 테스트 (`*.test.tsx`)

컴포넌트가 주어진 상태에서 **올바른 UI를 표시하는지**만 검증한다.

- 초기 렌더링 시 필요한 요소가 화면에 존재하는가
- 사용자 인터랙션 후 UI가 올바르게 변경되는가
- 조건에 따라 요소가 표시/숨김 처리되는가
- 로딩, 에러, 빈 상태 등 각 상태에 맞는 UI가 표시되는가

**검증하지 않는 것:**
- 에러 메시지의 구체적인 텍스트 내용 (스키마 테스트의 책임)
- API 응답 데이터의 가공 로직 (유틸/훅 테스트의 책임)
- 비즈니스 규칙 자체의 정합성 (스키마/로직 테스트의 책임)

### 스키마/유효성 검증 테스트 (`*.test.ts`)

valibot, zod 등 스키마 라이브러리의 **검증 규칙과 에러 메시지**를 검증한다.

- 유효한 입력이 통과하는가
- 각 규칙 위반 시 올바른 에러 메시지가 반환되는가
- 경계값(최소/최대 길이, 범위)이 정확히 동작하는가

### 커스텀 훅 테스트 (`*.test.ts`)

훅의 **상태 변화와 반환값**을 검증한다.

- 초기 상태가 올바른가
- 액션 호출 후 상태가 올바르게 변경되는가
- 비동기 처리(로딩, 성공, 실패) 흐름이 올바른가

### 유틸/헬퍼 테스트 (`*.test.ts`)

순수 함수의 **입력 → 출력**을 검증한다.

- 정상 입력에 대한 기대 출력
- 경계값, 빈 값, 잘못된 타입 등 엣지 케이스

### 책임 분리 예시: 회원가입 폼

```
// schemas/signup.test.ts — 스키마 테스트
describe('SignupSchema', () => {
  it('이메일 형식이 올바르지 않으면 "올바른 이메일을 입력해주세요" 에러를 반환한다')
  it('비밀번호가 8자 미만이면 "비밀번호는 8자 이상이어야 합니다" 에러를 반환한다')
  it('유효한 입력이면 에러 없이 통과한다')
})

// components/SignupForm.test.tsx — 컴포넌트 렌더링 테스트
describe('SignupForm', () => {
  it('이메일, 비밀번호 입력 필드와 가입 버튼이 표시된다')
  it('유효하지 않은 값을 제출하면 에러 메시지 영역이 표시된다')
  it('가입 버튼을 클릭하면 제출 처리된다')
  it('제출 중에는 버튼이 비활성화되고 로딩 상태가 표시된다')
})
```

컴포넌트 테스트에서 `"올바른 이메일을 입력해주세요"`라는 특정 메시지 텍스트를 검증하지 않는다.
메시지 텍스트가 변경되면 스키마 테스트만 수정하면 되고, 컴포넌트 테스트는 영향받지 않는다.

## 모키스트 vs 클래시스트 전략

테스트 대상의 성격에 따라 적절한 방식을 선택한다.

### 클래시스트 (Classicist) — 기본 전략

실제 구현에 가깝게 테스트한다.

- 컴포넌트 렌더링 + 사용자 인터랙션 테스트
- 상태 변경에 따른 UI 반영 테스트
- 자식 컴포넌트를 mock하지 않고 함께 렌더링한다

### 모키스트 (Mockist) — 외부 의존성이 있을 때

- API 호출: `msw` 또는 `vi.mock`으로 네트워크 요청을 mock
- 라우터/네비게이션: 라우터 동작을 mock
- 타이머/날짜: `vi.useFakeTimers()`로 시간 의존 로직을 제어
- 복잡한 전역 상태: 외부 store를 mock하여 특정 상태를 주입

### 판단 기준

| 테스트 대상 | 전략 |
|------------|------|
| 컴포넌트 자체의 렌더링과 인터랙션 | 클래시스트 |
| 외부 시스템 (API, 라우터, 브라우저 API) 경계 | 모키스트 |
| 판단이 어려운 경우 | 클래시스트 우선, 느리거나 불안정하면 모키스트로 전환 |
