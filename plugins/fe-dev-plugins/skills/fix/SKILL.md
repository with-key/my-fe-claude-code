# /fix — 버그 진단 및 기획 변경 수정 방안 제시

> 트리거 키워드: 버그, bug, 에러, error, 오류, 안 됨, 깨짐, 동작 안 함, fix, 수정, 디버깅, 변경, 기획 변경, 스펙 변경

## 사용법

사용자가 전달한 증상 또는 변경 설명: `$ARGUMENTS`

### 동작

**`/fix <버그 증상 또는 변경 설명>`** — 버그를 진단하거나 기획 변경에 따른 수정 방안을 제시한다.

예시:
```
# 버그 수정 모드
/fix 장바구니에서 수량 변경 시 총 금액이 업데이트되지 않음
/fix TypeError: Cannot read properties of undefined (reading 'map')
/fix 로그인 후 리다이렉트가 무한 루프에 빠짐
/fix 모바일에서 드롭다운 메뉴가 화면 밖으로 넘침

# 기획 변경 모드
/fix 최소 주문 금액이 10,000원에서 15,000원으로 변경됨
/fix 회원 등급별 할인율 정책이 변경됨 — 골드 10%→15%, 실버 5%→8%
/fix 상품 상세 페이지에 '사이즈 가이드' 탭 추가 필요
```

## 진단 프로세스

### Step 0: 모드 판별

사용자 입력을 분석하여 모드를 결정한다:

- **버그 수정 모드**: 에러 메시지, "안 됨", "깨짐", "에러", "오류", 스택 트레이스 등 결함 키워드가 포함된 경우
- **기획 변경 모드**: "변경", "추가", "제거", "정책 변경", 비즈니스 규칙 변경 키워드가 포함된 경우
- **모호한 경우**: 사용자에게 "버그 수정인지, 기획 변경에 따른 수정인지" 확인

### 1단계: 증상/변경 분석

**버그 수정 모드:**
- **에러 유형**: 런타임 에러, 렌더링 이슈, 상태 관리, 스타일링, 네트워크 등
- **키워드 추출**: 컴포넌트명, 함수명, 에러 메시지, 화면명 등 코드 검색에 활용할 단서
- **재현 조건 추정**: 증상에서 유추할 수 있는 트리거 조건

**기획 변경 모드:**
- **변경 전후 비즈니스 규칙 추출**: 현재 규칙과 변경될 규칙을 명시
- **키워드 추출**: 변경 대상 도메인 용어, 상수값, 정책명 등 코드 검색 단서
- **변경 범위 추정**: 해당 규칙이 적용되는 기능/화면 추정

### 2단계: 관련 코드 탐색

추출한 키워드로 코드베이스를 체계적으로 탐색한다:

1. **직접 연관 파일**: 에러 메시지, 컴포넌트명, 함수명, 상수값으로 검색
2. **의존성 추적**: import/export 체인을 따라 관련 모듈 파악
3. **상태 흐름 추적**: 상태 정의 → 변경 → 소비 경로를 추적
4. **테스트 파일 확인**: 관련 테스트가 존재하는지, 해당 케이스를 커버하는지 확인

### 3단계: 원인/변경 필요점 분석

**버그 수정 모드** — 근본 원인 규명:
- 로직 오류 (조건문, 계산, 순서)
- 타입/데이터 불일치 (null/undefined, 잘못된 프로퍼티 접근)
- 상태 관리 문제 (경쟁 조건, 스테일 클로저, 잘못된 의존성 배열)
- 비동기 처리 문제 (미처리 에러, 타이밍 이슈)
- CSS/레이아웃 문제 (z-index, overflow, 반응형)
- 환경/설정 문제 (빌드 설정, 환경 변수)

원인이 여러 개일 수 있으면 가능성이 높은 순서대로 나열한다.

**기획 변경 모드** — 변경 필요점 분석:
- 현재 코드가 이전 비즈니스 규칙을 어떻게 구현하고 있는지 파악
- 새 규칙 적용을 위해 어떤 부분을 수정해야 하는지 도출
- 상수, 조건문, 검증 로직, UI 표시 등 변경 포인트 나열

### 4단계: 사이드 이펙트 분석

수정 시 영향받을 수 있는 범위를 파악한다:

- **직접 영향**: 같은 모듈을 사용하는 다른 기능
- **간접 영향**: import 체인으로 연결된 파일
- **테스트 영향**: 수정으로 인해 깨질 수 있는 기존 테스트
- **UI 영향**: 레이아웃이나 스타일 변경이 다른 화면에 미치는 영향

### 5단계: 수정 방안 제시

**버그 수정 모드:**
- **수정 대상 파일과 위치**: 어떤 파일의 어떤 부분을 수정해야 하는지
- **수정 방향**: 어떻게 수정해야 하는지 코드 예시와 함께 설명
- **검증 방법**: 수정 후 어떻게 확인할 수 있는지 (테스트 실행, 수동 확인 포인트)
- **예방 조치**: 같은 유형의 버그가 재발하지 않도록 하는 방법 (테스트 추가, 타입 강화 등)

**기획 변경 모드:**
- **수정 대상 파일과 위치**: "현재 → 변경" 형식으로 구체적 수정 내용 제시
- **수정 순서**: 의존 관계를 고려한 권장 수정 순서
- **검증 방법**: 변경 후 확인해야 할 시나리오 목록
- **테스트 업데이트**: 수정해야 할 테스트 케이스와 추가할 테스트 케이스

## 주의사항
- 코드를 직접 수정하지 않는다. 진단 결과와 수정 방안만 제시한다.
- 원인이 불확실한 경우 가능성과 함께 명시하고, 추가 확인이 필요한 사항을 안내한다.
- 에러 메시지가 포함된 경우 스택 트레이스를 기반으로 분석한다.
- CLAUDE.md가 있으면 프로젝트 컨벤션 기준으로, 없으면 일반적인 프론트엔드 베스트 프랙티스 기준으로 분석한다.
