# /spec — 기획 문서 분석 및 프론트엔드 기능명세서 생성

> 트리거 키워드: 기획, PRD, 명세, spec, 기능명세서, 정책, 요구사항, 문서 분석

## 사용법

사용자가 전달한 액션: `$ARGUMENTS`

### 동작

**`/spec <파일경로>`** — 기획 문서를 분석하고 프론트엔드 기능명세서를 생성한다.

#### 1단계: 기획 문서 분석

시니어 프론트엔드 개발자 관점에서 기획 문서를 다각도로 분석한다.

- **요구사항 추출**: 기능 요구사항, 비즈니스 규칙, 사용자 플로우를 정리한다.
- **엣지 케이스 검토**: 기획서에 명시되지 않은 예외 상황, 경계값, 비정상 플로우를 식별한다.
- **기존 구현 상충 검토**: 프로젝트의 기존 코드와 상충되거나 충돌할 수 있는 부분을 확인한다.
- **기술적 제약 분석**: 프론트엔드 구현 시 고려해야 할 기술적 제약사항을 파악한다.

#### 2단계: 프론트엔드 기능명세서 생성

분석 결과를 바탕으로 `docs/specs/<기능명>.md`에 기능명세서를 작성한다.

기능명세서에는 다음 내용을 포함한다:

- **기능 개요**: 기능의 목적과 범위
- **화면별 상세 명세**: 화면 단위로 UI 요소, 상태, 인터랙션을 정의한다.
- **상태 관리**: 각 화면/컴포넌트의 상태 흐름과 전이 조건을 정의한다.
- **데이터 흐름**: 필요한 API 호출, 요청/응답 데이터 구조, 로딩/에러 처리를 정의한다.
- **엣지 케이스 및 예외 처리**: 분석에서 식별한 엣지 케이스와 대응 방안을 명시한다.
- **기획 피드백**: 기획자에게 확인이 필요한 질문 사항, 모호한 정책, 보완이 필요한 부분을 정리한다.

#### 3단계: 결과 출력

- 분석 요약과 주요 발견 사항을 출력한다.
- 기획 피드백 항목이 있으면 우선순위별로 강조 표시한다.

## 산출물 저장 위치

프로젝트 루트 기준:
```
docs/specs/   ← 프론트엔드 기능명세서
```

## 주의사항
- 기획 문서에 명시된 내용을 기반으로 작성한다. 추측으로 요구사항을 추가하지 않는다.
- 누락되거나 모호한 정보는 기획 피드백 섹션에 질문 형태로 정리한다.
- 기존 프로젝트 코드를 반드시 탐색하여 상충 여부를 확인한다.
- 파일명은 kebab-case를 사용한다.
